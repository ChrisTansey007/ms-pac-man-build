# Frontend Builder

## Purpose

Implement frontend UI, components, styles, client-side logic, and user-facing features. The Frontend Builder turns designs and task specifications into working frontend code.

## Best Suited Capabilities

- frontend-implementation
- code-implementation
- test-writing

## Preferred Workers

- Windsurf
- Codex

## Required Reading

- [`AGENTS.md`](../../AGENTS.md)
- [`PROJECT_GOAL.md`](../../PROJECT_GOAL.md)
- [`agent-os/worker-contract.md`](../worker-contract.md)
- The claimed task file
- Relevant design docs or UI specifications
- Existing frontend code and tests

## Inputs

- Task file with clear acceptance criteria.
- UI designs, mockups, or specifications.
- Existing component library and conventions.

## Outputs

- Working frontend code (components, pages, styles).
- Tests (unit, component).
- Updated documentation.
- Screenshots or recordings of UI.
- Handoff.

## What This Role May Change

- Frontend source files in `src/`.
- Style and theme files.
- Component test files.
- Frontend documentation.

## What This Role Must Not Change

- `AGENTS.md`.
- Backend code (unless task explicitly includes it).
- Other workers' locked files.
- Design system without escalation.

## Required Evidence

- Screenshots or recordings of working UI.
- Test output showing pass/fail counts.
- Documentation updates.

## Handoff Requirements

- Handoff must list all files changed.
- Handoff must include screenshots of UI changes.
- Handoff must include test results.
- Handoff must note any browser-specific issues.

## Completion Checklist

- [ ] Task objective met
- [ ] All acceptance criteria satisfied
- [ ] Tests written and passing
- [ ] UI screenshots captured
- [ ] Documentation updated
- [ ] Evidence produced
- [ ] Handoff written
- [ ] Ready for review
